# Promise 视频文本生成设计

## 背景

当前项目基于 `remotion` 与 `react`，已有的内容明显偏向“编程知识可视化讲解视频”路线，已经存在诸如原型链、事件循环等主题的动画组件与分段式场景组织方式。

因此，这个项目最适合承载以下类型的视频：

- 编程知识讲解
- 抽象机制可视化
- 代码执行流程演示
- 概念对比类教学短片

不建议将其作为优先方向的类型：

- 真人剧情
- 重素材剪辑
- 电影感广告片
- 强依赖实拍镜头的内容

## 本次目标

设计一套“仅通过文本输入，就能让项目生成对应教学视频”的协作方式。

首个样例视频主题为：

- 主题：`Promise`
- 受众：前端初学者
- 时长：约 5 分钟
- 风格：问题背景 + 动画解释 + 小例子
- 配音风格：老师主讲 + 关键处博主化强调
- 成片要求：字幕、配音、BGM 均为必选输出

## 推荐工作流

推荐采用两段式生成，而不是直接让自然语言一步到位生成最终视频：

1. 用户提供自然语言文本
2. 助手将文本整理为结构化视频规格
3. Remotion 组件读取规格并渲染视频

即：

`文本需求 -> 视频规格 -> Remotion 渲染`

这样做的好处：

- 用户输入门槛低
- 助手可以补全缺失信息
- 视频结构更稳定，可重复修改
- 后续可以沉淀为模板化生产方式

## 推荐成片结构

对于初学者的 5 分钟 `Promise` 视频，推荐使用混合型结构：

1. 开场问题：为什么回调会让代码变难写
2. 概念引入：Promise 是什么
3. 状态解释：`pending` / `fulfilled` / `rejected`
4. 行为解释：`resolve` / `reject`
5. 使用方式：`then` / `catch` / `finally`
6. 链式调用：为什么可以缓解回调地狱
7. 小例子：`setTimeout` 或模拟请求
8. 常见错误：例如 `then` 中遗漏 `return`
9. 总结：让观众记住的 2-3 个结论

## 成片交付物

每次接收文本需求后，助手应输出以下内容：

- 讲稿：逐段旁白文案
- 字幕：与旁白对齐的字幕文本
- 分镜：每段画面、镜头与转场说明
- 动画说明：每段需要什么视觉效果与状态变化
- 配音说明：语气、停顿、重音、节奏建议
- BGM 方案：推荐风格、情绪起伏、音量压制建议
- 视频规格：供 Remotion 读取的结构化数据

## 输入要求

### 最低配输入

只提供主题也可以启动：

```txt
主题：Promise
```

### 推荐输入

推荐用户至少提供以下信息：

```txt
主题：Promise
受众：前端初学者
时长：5分钟
必讲：为什么需要 Promise、三种状态、then/catch、链式调用
不讲：规范细节、微任务深挖
风格：通俗 + 动画解释
例子：setTimeout 模拟异步请求
BGM：默认
```

### 可选补充

用户还可以进一步提供：

- 自己的讲解提纲或知识点
- 希望模仿的讲解风格
- 自己的 BGM
- 不想出现的术语或展开内容
- 这条视频最想让观众记住的结论

## 协作方式

推荐采用以下协作方式：

- 用户以“主题 + 必讲点”为主输入
- 助手先给出第一版结构与脚本
- 用户逐段确认或修改
- 确认后再进入 Remotion 组件与渲染层实现

这是一种 `A + D` 的混合方式：

- A：助手主导补全内容
- D：双方逐段打磨

## BGM 规则

### 默认方案

若用户未提供 BGM，则使用默认教学视频音乐策略：

- 轻科技感
- 节奏稳定
- 不抢旁白
- 开场略强，中段压低，结尾轻微抬起

### 用户自带音乐

若用户提供 BGM，则后续需要：

- 根据音乐段落重排镜头节奏
- 调整字幕切换速度
- 在旁白密集处降低音乐存在感

## 结论

该项目最适合做“知识可视化”类编程教学视频。

要实现“纯文本驱动的视频生成”，最佳方案不是直接用一句话生成最终视频，而是先把文本沉淀成结构化视频规格，再由 Remotion 模板负责稳定渲染。

对当前项目而言，这是最稳妥、最可扩展、也最适合长期复用的工作模式。
